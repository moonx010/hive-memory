# Requirements: calendar-connector

## Functional Requirements

### REQ-CAL-01: Google Calendar API Integration

- MUST fetch events via Google Calendar API v3 using pre-generated credentials (service account JSON or OAuth2 token JSON).
- MUST support both credential modes:
  - **Service account**: read JSON key file → sign JWT (RS256 via `node:crypto`) → exchange for access token via `https://oauth2.googleapis.com/token`.
  - **OAuth2 token**: read `token.json` with `access_token` / `refresh_token`; refresh via `https://oauth2.googleapis.com/token` when `expires_in` is past.
- MUST handle paginated results using the `pageToken` field from the Calendar API response (loop until no `nextPageToken`).
- MUST respect rate limits: on 403 (User Rate Limit Exceeded) or 429 response, apply exponential backoff (initial 1s, max 32s, jitter ±20%) and retry up to 5 times before surfacing error.
- MUST NOT import the `googleapis` npm package. All HTTP calls use Node.js built-in `fetch`.

### REQ-CAL-02: Entity Production

- MUST produce `event` entities for all calendar events retrieved.
- MUST promote an event to a `meeting` entity when either of these conditions holds:
  - `attendees` array length >= 3 (including organizer), OR
  - `conferenceData` field is present and non-null.
- MUST produce `person` entities for unique attendees (deduplicated by `email`). Each attendee email maps to exactly one `person` entity per sync run.
- MUST set `source.externalId` using these formats:
  - Events: `"gcal:event:{calendarId}:{eventId}"`
  - Persons: `"gcal:person:{email}"`
- MUST set `meetingType` attribute on `meeting` entities:
  - 2 attendees → `"one-on-one"`
  - 3–8 attendees → `"small-group"`
  - 9+ attendees → `"large-meeting"`
- MUST populate `attributes.conferenceUrl` from `conferenceData.entryPoints[0].uri` if present.

### REQ-CAL-03: Synapse Creation

- MUST create `"attended"` synapses from each `person` entity to each `meeting` entity they attended (using `responseStatus != "declined"`).
- MUST create `"temporal"` synapses between recurring event instances that share the same `recurringEventId`, linking them in chronological order (earlier → later).
- MUST NOT create duplicate synapses on incremental re-sync. Check for existing synapse by `(source_id, target_id, axon_type)` before inserting.

### REQ-CAL-04: Incremental Sync

- MUST track sync cursor as ISO8601 timestamp stored in the `connectors` table `last_sync` column.
- MUST pass cursor as the `updatedMin` parameter to `events.list` on incremental sync runs.
- MUST upsert existing entities on re-sync using `source_external_id` as the unique key — no duplicate entities.
- MUST handle cancelled events: when `event.status === "cancelled"`, set the entity's `status` column to `"archived"` (update existing row, do not create a new entity).
- MUST default to fetching events from `timeMin = now() - 90 days` on first full sync when no cursor is present.

### REQ-CAL-05: Configuration

- MUST read credentials file path from `GOOGLE_CALENDAR_CREDENTIALS` environment variable.
- MUST read optional OAuth2 token file path from `GOOGLE_CALENDAR_TOKEN` environment variable.
- MUST read calendar IDs from `GOOGLE_CALENDAR_IDS` environment variable as a comma-separated list; default to `["primary"]` if not set.
- MUST return `false` from `isConfigured()` if:
  - `GOOGLE_CALENDAR_CREDENTIALS` is not set, OR
  - The file at the specified path does not exist or is not readable.
- MUST read calendar IDs at sync time, not at construction time (to support env-var override in tests).

### REQ-CAL-06: Error Handling

- MUST emit a structured error log (with `calendarId`, `eventId`, HTTP status) for individual event failures, and continue processing remaining events.
- MUST surface auth errors as a thrown `Error` with message `"CalendarConnector: auth failed — {reason}"`.
- MUST surface API errors (non-200, non-retryable) as `Error` with HTTP status code included.

## Non-Functional Requirements

- MUST NOT add any npm dependencies (use `fetch`, `node:crypto` from Node.js stdlib).
- MUST follow the existing `ConnectorPlugin` interface defined in `src/connectors/types.ts`.
- MUST follow code patterns established in `src/connectors/github.ts` (async generators, retry helper, transform method).
- MUST complete a full sync of 1,000 events in under 60 seconds on a standard developer machine (excluding network latency).
- MUST be registered in `src/store.ts` only when `GOOGLE_CALENDAR_CREDENTIALS` env var is set (guard identical to GitHub/Slack connector registration pattern).

## Out-of-Scope Constraints

- OAuth2 browser authorization flow is NOT in scope. Token must be pre-generated by the user.
- Calendar API write operations are NOT in scope.
- Google Calendar push notification webhooks are NOT in scope.
- Non-Google calendar sources (iCal, Exchange, CalDAV) are NOT in scope.
